## Summary

`programs/roundfi-yield-kamino/src/lib.rs:84` pinned the Kamino Lend
program ID as a base58 literal one character off from the canonical
mainnet value. The typo compiled cleanly, passed every unit test,
passed every bankrun spec that didn't load Kamino's real bytecode, and
would have failed only at first contact with mainnet — at the highest-
cost detection moment in the lifecycle.

```rust
// Before SEV-040 fix:
pub const KAMINO_LEND_PROGRAM_ID: Pubkey =
    anchor_lang::pubkey!("KLend2g3cPP7fffoy8q1mQqGKjrxjC8boSyAYavgmjD");
//                              ^^ — positions 10-11

// After fix:
pub const KAMINO_LEND_PROGRAM_ID: Pubkey =
    anchor_lang::pubkey!("KLend2g3cP87fffoy8q1mQqGKjrxjC8boSyAYavgmjD");
//                              ^^ — canonical (P8)
```

The fix is a 1-character edit. The post-mortem is about why one
character mattered for three months.

## Root cause analysis (5 whys)

1. **Why did SEV-040 ship?** The constant `KAMINO_LEND_PROGRAM_ID` was
   declared with a 1-character typo.
2. **Why didn't the compiler catch it?** `anchor_lang::pubkey!()` is a
   compile-time macro that validates base58 syntax (no invalid
   characters, correct length, valid encoding), but has no notion of
   "canonical" — any syntactically-valid base58 string is accepted as
   `Pubkey`. The same macro would have accepted `KLend2g3cPP…` and
   `KLend2g3cP8…` equally.
3. **Why didn't the test suite catch it?** Existing tests in
   `programs/roundfi-yield-kamino/src/lib.rs::tests` pinned the
   instruction discriminators (`anchor_ix_discriminator("deposit")`,
   `…("withdraw")`, `…("harvest")`) and asserted they were distinct
   and non-zero. The program ID itself was not pinned by any test.
   Discriminator tests verify "we compute the same hash that Anchor
   uses" — orthogonal to "we point at the real Kamino program."
4. **Why didn't integration testing catch it?** The localnet and
   bankrun specs both deployed `roundfi-yield-mock` as the adapter,
   not `roundfi-yield-kamino`. The mock adapter has its own program
   ID and never invokes the `address = KAMINO_LEND_PROGRAM_ID`
   constraint. `roundfi-yield-kamino` was deployable in isolation but
   never end-to-end tested against the real Kamino program until the
   spike — which is itself the activity that surfaced the bug.
5. **Why didn't code review catch it?** Code reviewers (humans)
   eyeballing 44-character base58 strings cannot reliably distinguish
   `KLend2g3cPP…` from `KLend2g3cP8…`. The visual signal-to-noise is
   close to zero for this class of literal. The review process did
   not require citing an external canonical source for the constant
   or attaching a parity test in the same PR.

## What surfaced it

The Kamino bankrun-clone spike, planned to validate the Kamino CPI
mechanics before canary-mainnet. The spike's Phase 1 step "load real
Kamino bytecode" required identifying the program address on mainnet
to clone. The operator opened the canonical Kamino mainnet program ID
in the [Kamino-Finance/klend `declare_id!`](https://github.com/Kamino-Finance/klend)
source in one tab, the RoundFi constant in another, and read them
side-by-side as a sanity check before issuing `solana program dump`.
The mismatch was caught in that comparison, before bankrun even
started.

This is consistent with the deepest lesson from this post-mortem:
**typos at this character density survive every automated check that
doesn't compare against a canonical authority.** Manual side-by-side
comparison eventually catches them — but only when the workflow
forces the comparison.

## What the fix added

1. **1-character source edit** (`cad9ca4`): `PP` → `P8` in
   `programs/roundfi-yield-kamino/src/lib.rs:84`.

2. **New unit test** (`kamino_lend_program_id_matches_canonical` in
   the same file's `#[cfg(test)] mod tests` block):

   ```rust
   #[test]
   fn kamino_lend_program_id_matches_canonical() {
       // Asserts the literal string against the canonical mainnet
       // value. Sub-second; runs on every `cargo test` invocation.
       assert_eq!(
           KAMINO_LEND_PROGRAM_ID.to_string(),
           "KLend2g3cP87fffoy8q1mQqGKjrxjC8boSyAYavgmjD",
       );
   }
   ```

   This is the surface-level fix. The next section is the bigger one.

## Methodology lesson and downstream propagation

**The deeper lesson.** Every "pinned external-protocol constant" in
the codebase — program IDs, byte offsets, PDA seeds, instruction
discriminators referring to upstream protocols — is in the same class
as `KAMINO_LEND_PROGRAM_ID`. Each:

- Compiles against a syntactic check, not a semantic one.
- Survives unit tests that exercise the constant's _use_ but not its
  _value_.
- Survives integration tests that never run the constant against the
  real external system.
- Cannot reliably be reviewed visually.

The required response shape is consistent: **every such constant
needs an automated assertion that ties it to an authoritative source,
and that assertion lives in the same PR as the constant.**

**Propagation across the codebase (May 2026 follow-ups).** Three
audit waves applied the SEV-040 lesson to adjacent surfaces, finding
two more same-class bugs that would have shipped to canary undetected:

Three SEVs — one Critical (SEV-042), one same-family follow-up

priming pattern. The pattern is now the standard response to any "X
constant is pinned" finding: **audit the surface, add a coupling test,
log the finding even if no current bug exists.**

## Process gap (not just a test gap)

The deeper issue with SEV-040 is not "we needed one more unit test."
It is that the code-review process for pinned external-protocol
constants treated them as text-equivalent to any other base58 string.
The PR introducing the constant did not:

1. Cite the canonical source (Kamino's `declare_id!`, an IDL file, a
   block explorer link, or similar) in the commit message or comment.
2. Include a parity test in the same commit that verified the
   constant against an authoritative reference.
3. Force the reviewer to perform the side-by-side comparison.

**New process rule (informal, but now applied).** Any commit that
adds or modifies a string/byte/offset constant referring to an
external system must include, in the same commit:

- A comment citing the authoritative source (a URL, a quoted
  `declare_id!`, a block explorer reference).
- An automated assertion (unit test / parity test / coupling test)
  that fails if the constant drifts from that source.

This rule is encoded in the audit doc's methodology notes and
demonstrated by the SEV-040, SEV-042, and SEV-043 fix patterns.

## What the formal Adevar audit can take from this

When the external audit engages, the reviewer can verify the team's
internal process response against the codebase pattern:

1. **Pattern match** — grep for `pubkey!("…")`, `b"…"` constants,
   byte-offset arithmetic, discriminator literals, and verify each
   has a sibling assertion in `#[cfg(test)] mod tests` or in a
   parity test file.

2. **Coupling-test coverage** — the parity test in
   `tests/parity.spec.ts` should cover every Rust ↔ TS shared
   constant. Gaps are findings of the SEV-043 class.

3. **Byte-offset stability** — anything reading raw account bytes
   off-chain should have a Rust-side size pin (see
   `state/config.rs::tests::protocol_config_size_pinned_for_hardening_script`
   for the canonical pattern).

4. **CPI account-list pinning** — every external-protocol CPI
   account-list construction should have a sibling cargo test that
   pins each slot's (pubkey, is_signer, is_writable) tuple against
   the canonical layout. Five surfaces are currently covered (Kamino
   deposit, Kamino redeem, adapter prelude, reputation attest, plus
   asymmetry guards).

The team's internal sweep (Pass-8, Pass-9) found Critical drift in
the hardening script and Medium drift in parity tests. **If the
external audit finds another instance of the same shape, the
methodology is in place to absorb it the same day.** If it finds
zero, the internal sweeps were thorough.
