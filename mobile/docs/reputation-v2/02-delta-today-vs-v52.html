Comparativo interativo entre o estado atual do RoundFi e as mudanças propostas na v5.2

Camada de eventos — como defaults e atrasos são registrados
Classificação de eventos
roundfi-reputation / settle_default
Hoje
3 valores
PAYMENT_MISSED · INFRA_FAILURE · VOLUNTARY_EXIT
Tudo que não é saída voluntária ou falha de infra vira PAYMENT_MISSED — sem gradação de gravidade. Um atraso de 2h e um abandono definitivo recebem o mesmo rótulo.
Com v5.2
11 variantes
PaymentOnTime · PaymentEarly · FrictionOperational · FrictionTemporal · LateBehavioral · TemporaryIncapacity · Default · BadFaith · CycleComplete · PoolComplete · Recovery
Cada evento carrega sua natureza. Score reflete comportamento real, não categorias brutas.
Cálculo do score
roundfi-reputation / src/math/
Hoje
Contadores simples
on_time_count · late_count · defaults como campos do ReputationProfile. Não há fórmula pública documentada. Score é derivado implicitamente dessas métricas sem lógica auditável externamente.
Com v5.2
4 funções puras
reliability() · punctuality() · commitment() · recovery() — funções determinísticas com pesos publicados em constants.rs. Qualquer terceiro roda localmente e chega no mesmo número.
Prova de fricção operacional
roundfi-reputation / friction_proof
Hoje
Não existe
Outage de RPC, congestão de rede ou bug de wallet resultam em INFRA_FAILURE sem verificação. Depende de atestação manual sem rastreabilidade on-chain.
Com v5.2
FrictionProof on-chain
4 variantes verificáveis: OnChainOracle · FailedTransaction · ValidatorOutageWindow · GovernanceAttested. Cada prova é verificável por qualquer nó. Janela de 7 dias para submissão.
Escada de níveis
constants.rs / resolve_tier
Hoje
3 níveis (50/30/10%)
L1 Iniciante · L2 Comprovado · L3 Veterano. Documentado em todos os materiais institucionais. Critérios de promoção implementados mas sem fórmula determinística pública.
Com v5.2
4 níveis (50/25/10/3%)
Adiciona L4 Elite (≥8 pools, wallet≥2a, PoP ativo, 0 BadFaith). L2 muda de 30% para 25%. Função resolve_tier() determinística — promoção e rebaixamento automáticos.
Storage do histórico
programs/roundfi-reputation / state
Hoje
Contadores agregados
ReputationProfile armazena totais: completed_cycles · on_time_payments · late_payments · defaults. Eventos individuais via Attestation mas sem classificação granular.
Com v5.2
Evento por evento, append-only
BehavioralEvent por ciclo com delta_seconds · classification · friction_proof · classification_sealed_at. C2 imutável após 7 dias. Permite recomputar qualquer ponto histórico.
Consumo externo do score
programs/roundfi-reputation / query
Hoje
Não existe como CPI
Score existe na tela de reputação do front-end e como dado do indexer. Não há instrução on-chain que outros programas Solana possam chamar via CPI para verificar o tier de uma wallet em runtime.
Com v5.2
Score Reader Program
Instrução query_score() read-only, permissionless. Retorna ScoreSummary { tier, reliability, punctuality, is_stale }. Qualquer protocolo Solana pode verificar tier de uma wallet antes de admiti-la.
O que muda na experiência do usuário e parceiros
Usuário que pagou com 2h de atraso por congestão de RPC
Hoje: PAYMENT_MISSED — penalidade
v5.2: FrictionOperational — zero impacto no score
Usuário que pagou 1 dia atrasado por esquecimento
Hoje: mesmo rótulo que inadimplência
v5.2: FrictionTemporal — queda de 0.05 pt em janela de 50
Parceiro B2B que quer integrar o score
Hoje: precisa confiar no operador, sem verificação independente
v5.2: recomputa score localmente com código aberto, via CPI
Usuário avançado querendo atingir L4 Elite
Hoje: L3 Veterano é o teto (10% stake)
v5.2: L4 Elite com 3% stake para histórico longo e limpo
Protocolo externo filtrando membros antes de admitir em pool
Hoje: impossível via CPI — depende de API off-chain
v5.2: query_score(wallet) retorna tier em runtime on-chain
Usuário que quer contestar penalidade por falha de infra
Hoje: não há mecanismo formal de contestação
v5.2: 7 dias para submeter FrictionProof — reclassificação automática
Riscos que a v5.2 elimina ou reduz
Score não verificável por parceiro
ALTO
Hoje, um parceiro (CREDO, Huma Finance) que consome o score precisa confiar que o RoundFi calculou corretamente — black box. Isso é barreira de adoção para crédito real. Com v5.2, o parceiro baixa o código, roda as funções puras com o estado on-chain da wallet, e chega no mesmo número. Score vira dado verificável, não declaração de confiança.
Penalidade injusta por falha de infraestrutura
ALTO
Hoje, um outage de RPC ou congestão de Solana que impede o usuário de pagar resulta em PAYMENT_MISSED — mesmo rótulo de inadimplência. Com FrictionProof on-chain, o usuário apresenta o tx-hash da transação que falhou ou o registro do oráculo de congestão, e o evento é reclassificado automaticamente como FrictionOperational sem impacto no score.
Score inconsistente após mudança de constante
MÉDIO
Hoje não há mecanismo de versioning de constantes. Se o time ajustar pesos do score, TierAssignments antigos ficam desatualizados silenciosamente. Com constants_version: u8 no TierAssignment, qualquer consumer sabe se o tier está stale e precisa ser recalculado antes de ser usado.
Dados de devnet incompatíveis com nova arquitetura
MÉDIO
Os ciclos já rodados em devnet geraram dados com o schema de 3 valores. Sem a migration Prisma mapeando PAYMENT_MISSED → LateBehavioral etc., qualquer demo ou análise que use dados históricos fica inconsistente com o novo schema. A migration precisa acontecer antes de novos ciclos gerarem dados com o formato errado.
Crank silencia falhas de score
MÉDIO
O orchestrator atual nunca chama settle_default. Se o crank de produção chamar settle_default mas não chamar record_event imediatamente depois, o score da wallet fica desatualizado sem que ninguém perceba — o pool sabe do default, a reputação não. A sequência atômica settle→record no crank elimina esse gap.
O que a v5.2 preserva integralmente do estado atual
roundfi-core
pool lifecycle completo
Sem mudança
Pool creation · contribute · claim_payout · default settlement · Escape Valve · yield waterfall — intactos. A v5.2 adiciona uma CPI de roundfi-core para roundfi-reputation após settle_default, mas não altera nenhuma lógica do core.
Triple Shield
Seed Draw · Escrow · Solidarity Vault
Sem mudança
Toda a lógica econômica de proteção é preservada. A v5.2 não toca em stake amounts, waterfall ordering, ou Solidarity Vault logic.
Stress Lab
sdk/stressLab.ts
Sem mudança
Simulações econômicas independentes da lógica de score. O Stress Lab modela solvência do pool, não comportamento de reputação.
Escape Valve (NFT)
Metaplex Core position
Sem mudança
O mecanismo de venda de posição continua idêntico. Na v5.2, VOLUNTARY_EXIT continua não penalizando score — mapeado para sem evento negativo como já é hoje.
Yield adapters
yield-mock · yield-kamino
Sem mudança
Kamino CPI e mock adapter intactos. O score de reputação não interfere com yield routing.
Program IDs
devnet deployado
Depende da abordagem
Se a v5.2 for implementada como upgrade do programa existente (Hpo174...e9R2), os Program IDs são preservados. Se for um redeploy com nova lógica de account, novos IDs. A decisão de upgrade vs redeploy precisa ser tomada pela equipe antes de qualquer implementação.
